## 开始之前

使用本文档前，请确保：

### 安装方式（推荐）

全局安装 Tackle Harness（推荐）：

```bash
npm install -g tackle-harness
tackle-harness init
```

### 迁移旧项目

如果项目之前使用本地安装模式，可以迁移到全局模式：

```bash
tackle-harness migrate
```

这会删除本地的 `.claude/skills/` 和 `.claude/hooks/` 目录，改用全局安装的技能和钩子。

### 备选方式（本地安装）

如果无法使用全局安装，仍然可以使用本地模式：

```bash
npx tackle-harness init
```

> 本文中 `/skill-xxx` 是 Claude Code 的**斜杠命令（slash command）**，在 Claude Code 提示符中输入即可触发对应能力。你也可以用自然语言触发，如"创建工作包"、"批量执行"等。

> **注意**: 全局安装后，所有技能和钩子由 `tackle-harness` 包统一提供，项目不再需要本地 `.claude/skills/` 和 `.claude/hooks/` 目录。

### 核心理念速览

| 原则 | 说明 | 对应 Tackle 能力 |
|------|------|------------------|
| **文档优先** | 先定义清楚做什么，再动手写代码 | task-creator、split-work-package |
| **人工把关** | 关键节点强制等你说"继续" | human-checkpoint、hook-skill-gate |
| **并行加速** | 多个工作包同时分给多个 Agent | agent-dispatcher (Agent Teams) |
| **经验复用** | 每次任务完成后沉淀经验 | experience-logger、role-manager |
| **全程可追溯** | 每个状态变更都有记录 | progress-tracker、task.md |

> 不确定用哪个 Skill？跳转到 [第 7 节 Skill 速查手册](#7-skill-速查手册)

| 阶段 | 触发方式 | 你的输入 | AI 做什么 |
|------|----------|----------|-----------|

| P1 | 自动触发 | "继续" / "修改..." | 等你确认后才放行 |

> 让 AI 自动串联 P0→P1→P2→P3→P4 全流程。适合需求明确、不想手动逐步触发的场景。

### 流程图

```
  ┌──────────────────────────────────────────────────────────────────┐
  │  前置条件: task.md 中有多个 📋 状态的 WP                          │
  └────────────────────────┬─────────────────────────────────────────┘

  └──────────────────────────────────────────────────────────────────┘
```

### 并行调度关键规则

```
  ┌─────────────────────────────────────────────────────────┐
  │  Agent Dispatcher 调度规则                                │
  │                                                          │
  │  1:1 绑定  ──  1 个 WP = 1 个 Teamee，完成即销毁         │
  │  并发上限  ──  默认最多 6 个 Teamee 同时运行（建议 ≤ 4）   │
  │  依赖调度  ──  blockedBy 依赖满足后才创建 Teamee          │
  │  记忆注入  ──  根据角色自动注入历史经验                    │
  │  强制清理  ──  完成后必须 TeamDelete，不可跳过            │
  │  循环检测  ──  依赖图存在环时报错终止                     │
  └─────────────────────────────────────────────────────────┘
```

### 实际操作示例

```
# 你的典型操作序列

1. "批量创建以下任务: ① 用户认证模块 ② 数据缓存层 ③ API 网关"
   → /skill-batch-task-creator 进入 Plan 模式
   → 生成 WP-036, WP-037, WP-038 文档

2. 审核确认 → "继续"

3. "批量执行 WP-036 到 WP-038"
   → /skill-agent-dispatcher
   → Lead Agent 分析依赖，创建 Team，并行调度

4. 等待全部完成（自动轮询 TaskList）

5. /skill-checklist     → 质量检查
6. /skill-completion-report → 生成完成报告
```

### 快速修复 Checklist

| 检查项 | 确认 |
|--------|------|

## 8. 最佳实践与避坑指南

### 工作包拆分建议

```
  ┌─────────────────────────────────────────────────────────────┐
  │  拆分粒度决策树                                              │
  │                                                              │
  │  影响文件数 ≤ 2 且无新接口?                                   │
  │  ├── 是 → simple 模式，不拆分                                │
  │  └── 否 → 影响模块数 > 3?                                    │
  │           ├── 是 → fine-grained，多模块并行                   │
  │           └── 否 → standard，4 阶段顺序执行                   │
  │                                                              │
  │  实际经验:                                                   │
  │  ├── 拆太细 → Agent 调度开销大于收益，单文件修复杂糅碎任务     │
  │  ├── 拆太粗 → 单个 Agent 上下文爆炸，容易遗漏                │
  │  └── 最佳 → 每个 WP 聚焦一个独立模块/功能点，预估 ≤ 30 分钟  │
  └─────────────────────────────────────────────────────────────┘
```

### 并行执行注意事项

| 事项 | 说明 |
|------|------|
| **并发上限** | 默认 ≤ 6 个 Teamee（建议控制在 4 以内避免资源竞争） |
| **1:1 绑定** | 严格 1 个 WP 对应 1 个 Teamee，完成即销毁 |
| **依赖顺序** | blockedBy 确保前置 WP 完成后才启动后续 |
| **强制清理** | 每次执行完必须 TeamDelete，不做资源池化 |
| **npm 发布** | 你有 2FA，涉及 npm publish 需手动处理 OTP |

### 常见问题

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| Plan 模式无法生成 WP | 需求描述不够具体 | 补充：涉及哪些文件、什么功能、预期行为 |
| Agent 执行偏离需求 | WP 文档定义模糊 | 重新审查 WP 文档，增加验收标准 |
| 团队资源未清理 | TeamDelete 失败 | `/skill-team-cleanup` 手动清理孤立团队 |
| skill-gate 阻止编辑 | 状态处于 "waiting" | 先通过 human-checkpoint 确认 |
| 角色匹配不精确 | role-registry 关键词不足 | `/skill-role-manager` 更新角色定义 |
| 会话中断后如何继续 | Claude Code 会话关闭丢失上下文 | 重新开对话 → `/skill-progress-tracker` 读取进度 → 查看 task.md 确认已完成项 → 从未完成的 WP 继续 |

### 一日工作流 Checklist

```
  晨间:
  □ /skill-progress-tracker 读取昨日进度
  □ 确认今日待办 WP 列表
  □ 对新需求执行 /skill-task-creator 或 /skill-batch-task-creator

  工作中:
  □ 人工审核确认后才放行 (P1)
  □ /skill-agent-dispatcher 并行执行已审核的 WP
  □ 完成后执行 /skill-checklist 质量检查

  收工前:
  □ /skill-completion-report 生成当日完成报告
  □ /skill-experience-logger 记录今日经验
  □ /skill-progress-tracker 保存进度快照
  □ 确认 task.md 状态已更新
```

### 守护进程（Watchdog）

当需要长时间运行的后台监控时（如持续编译、测试监听、进程守护）：

- 启动: `/skill-watchdog-manager` 或 "启动守护进程"
- 查看状态: "查看守护进程状态"
- 停止: "停止守护进程"

典型场景: 大型项目构建时启动 watchdog 监控编译进程，完成后自动通知。
